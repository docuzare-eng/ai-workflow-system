## 阶段详解

### Stage 0：可行性验证 🔍

**负责角色**: R0 可行性分析师  
**推荐工具**: Claude.ai (Opus)  
**核心问题**: 这个项目值得做吗？

**主要活动**:
- 市场调研：有没有人真的需要这个？
- 技术预研：关键技术难点是否可行？
- 资源评估：时间、精力、预算够吗？
- 机会成本分析：做这个会放弃什么？

**产出物**:
- `01-feasibility/FEASIBILITY_REPORT.md`
- `01-feasibility/MARKET_RESEARCH.md`
- `01-feasibility/RISK_ASSESSMENT.md`

**门禁标准**:
- ✅ 明确的目标用户和需求
- ✅ 关键技术难点有初步验证
- ✅ 有清晰的"不做"的边界

**典型耗时**: 1-3 天

---

### Stage 1：构想打磨 💡

**负责角色**: R1 产品构想师  
**推荐工具**: Claude.ai (Opus)  
**核心问题**: 这个产品究竟是什么？

**主要活动**:
- 苏格拉底式对话：反复追问想法
- 核心价值主张提炼
- MVP 边界划定
- 长期愿景勾画

**产出物**:
- `02-vision/VISION.md`

**门禁标准**:
- ✅ 能用 2-3 句话清晰描述产品
- ✅ MVP 功能清单明确
- ✅ 差异化定位清楚

**典型耗时**: 半天到 1 天

---

### Stage 2：需求规格化 📋

**负责角色**: R2 需求分析师  
**推荐工具**: Claude.ai (Opus)  
**核心问题**: 具体要实现什么？

**主要活动**:
- 撰写 PRD（产品需求文档）
- 用户故事拆解（带优先级 P0/P1/P2）
- 用户旅程图绘制
- 边界情况识别

**产出物**:
- `03-requirements/REQUIREMENTS.md`
- `03-requirements/USER_STORIES.md`
- `03-requirements/USER_JOURNEYS.md`

**门禁标准**:
- ✅ 每个功能能回答"用户点了什么看到什么"
- ✅ 用户故事有验收标准
- ✅ 识别出关键边界情况

**典型耗时**: 1-3 天

---

### Stage 3：架构设计 🏛️

**负责角色**: R3 架构师  
**推荐工具**: Claude.ai (Opus)  
**核心问题**: 用什么技术、怎么组织？

**主要活动**:
- 技术栈选型（2-3 方案对比）
- 架构模式选择（单体/微服务/Serverless）
- 数据库设计（ERD、表结构）
- API 设计
- 目录结构规划
- 安全设计
- 关键技术难点 POC

**产出物**:
- `04-architecture/ARCHITECTURE.md`
- `04-architecture/TECH_STACK.md`
- `04-architecture/DATABASE.md`
- `04-architecture/API_DESIGN.md`
- `04-architecture/DIRECTORY_STRUCTURE.md`
- `04-architecture/SECURITY_DESIGN.md`

**门禁标准**:
- ✅ 技术栈选型有理有据
- ✅ 数据库设计覆盖所有核心实体
- ✅ 高风险技术难点已验证
- ✅ 新人看文档能理解架构

**典型耗时**: 3-7 天（最重要的阶段之一）

---

### Stage 4：骨架搭建 🦴

**负责角色**: R4 工程骨架师  
**推荐工具**: Claude Code  
**核心问题**: 把架构变成可运行的项目骨架

**主要活动**:
- 初始化项目（脚手架）
- 配置工具链（ESLint、Prettier、TypeScript、测试框架）
- 创建目录结构
- 配置 CI/CD
- 编写 `CLAUDE.md`（Claude Code 的专属上下文）
- 配置 Git hooks

**产出物**:
- 可运行的空项目骨架
- `05-skeleton/SKELETON_REPORT.md`
- `05-skeleton/CLAUDE.md`

**门禁标准**:
- ✅ `npm run dev` 或等价命令能跑起来
- ✅ 代码规范、格式化、类型检查全部配好
- ✅ CI/CD 流水线跑通
- ✅ `CLAUDE.md` 描述了项目约定

**典型耗时**: 1-2 天

---

### Stage 5：功能开发 🔨

**负责角色**: R5 开发工程师（主力）+ R2/R3 配合规划  
**推荐工具**: Claude Code（实现）+ Claude.ai（规划）  
**核心问题**: 一个一个功能造出来

**主要活动**:（每个功能循环一次）
任务规划 (Claude.ai R2/R3 协作)
↓
实现 (Claude Code R5)
↓
代码审查 (R9)
↓
人工验证 (你)
↓
提交 + 文档更新
↓
进入下一个功能

**产出物**:
- 每个功能一个 `06-development/features/FEAT-XXX.md`
- 对应的代码 + 测试

**门禁标准**（单个功能）:
- ✅ 符合架构规范
- ✅ 有对应测试
- ✅ 代码审查通过
- ✅ 人工验证通过

**典型耗时**: 项目主体时间

---

### Stage 6：质量保障 🛡️

**负责角色**: R6 QA 工程师 + R11 风险侦察兵  
**推荐工具**: Claude.ai（设计）+ Claude Code（实现）  
**核心问题**: 项目真的能用、可靠、安全吗？

**主要活动**:
- 测试策略（单元/集成/E2E 分层覆盖）
- 安全审计（SQL 注入、XSS、CSRF、权限漏洞）
- 性能测试与优化
- 可观测性配置（日志、监控、告警）
- 依赖漏洞扫描

**产出物**:
- `07-quality/TEST_STRATEGY.md`
- `07-quality/SECURITY_AUDIT.md`
- `07-quality/PERFORMANCE_REPORT.md`

**门禁标准**:
- ✅ 核心功能测试覆盖率 > 70%
- ✅ 无已知高危安全漏洞
- ✅ 性能指标达到预算目标
- ✅ 监控告警可用

**典型耗时**: 占总时间 20-30%

---

### Stage 7：迭代演进 🚀

**负责角色**: R7 运维迭代师  
**推荐工具**: Claude.ai + Claude Code  
**核心问题**: 上线、观察、持续改进

**主要活动**:
- 部署上线
- 用户反馈收集
- 数据驱动迭代决策
- 技术债务管理
- 架构演进评估

**产出物**:
- `08-iteration/DEPLOYMENT.md`
- `08-iteration/MONITORING.md`
- `08-iteration/TECH_DEBT.md`

**门禁标准**:
- ✅ 部署流程可重复
- ✅ 用户反馈有收集渠道
- ✅ 技术债务有清单和优先级

**典型耗时**: 持续进行

## 阶段间的流动规则

### 正向流动（主路径）
`0 → 1 → 2 → 3 → 4 → 5 → 6 → 7`

### 允许的回退
- `3 → 2`：架构时发现需求不合理
- `5 → 3`：开发时发现架构缺陷
- `5 → 2`：开发时发现需求理解偏差
- `6 → 5`：质量问题需要代码修复
- `7 → 2/3`：用户反馈触发产品/架构调整

### 禁止的跳跃
- ❌ `0 → 4`：没想清楚就开始搭架子
- ❌ `2 → 5`：没架构就开发
- ❌ `5 → 7`：没质检就上线

## 贯穿性角色的介入时机

| 角色 | 介入时机 |
|------|---------|
| **R8 项目协调官** | 每个阶段开始/结束 + 每周巡检 |
| **R9 代码审查员** | 每次代码提交 |
| **R10 文档守护者** | 每个阶段结束 + 每两周巡检 |
| **R11 风险侦察兵** | Stage 0/3/6 重点介入 + 任意时刻发现风险 |

---

## 实操节奏建议

### 个人项目（1-2 人）
- Stage 0-3：1-2 周（慢工出细活）
- Stage 4：2-3 天
- Stage 5：项目主体，按功能迭代
- Stage 6-7：发布前集中 + 持续

### 小型项目（< 1 个月）
可以合并 Stage 0 和 Stage 1，但不要跳过。

### 大型项目（> 3 个月）
严格按流程走，Stage 3 可能需要 1-2 周深度设计。
