## 协调层角色

### 🎖️ R8 - 项目协调官（Project Coordinator / CTO）

**定位**: 整个项目的总指挥，你的"AI CTO"

**核心职责**:
- 审视每个阶段的产出质量
- 发现阶段间的不一致和矛盾
- 决定何时推进、何时回退
- 定期做项目健康度检查
- 协调其他角色的工作

**使用方式**:
- **独立长期窗口**：从项目开始到结束保持同一个对话
- 每个阶段结束时咨询它："我可以进入下一阶段吗？"
- 每周一次健康度巡检
- 遇到跨阶段问题时求助

**推荐模型**: Claude Opus（最强推理能力）

**Prompt 文件**: `prompts/cross-cutting/R8-project-coordinator.md`

## 阶段层角色

### 🔍 R0 - 可行性分析师（Feasibility Analyst）

**定位**: 项目的"守门人"，在投入前验证必要性

**核心职责**:
- 市场需求验证
- 技术可行性评估
- 资源和风险预估
- 给出"做 / 不做 / 换方向"的建议

**能做**:
- 结构化分析市场和竞争
- 识别关键技术风险
- 提供多种替代方案

**不做**:
- 不做具体实现
- 不写代码

**推荐模型**: Claude Opus

**Prompt 文件**: `prompts/stages/R0-feasibility-analyst.md`

---

### 💡 R1 - 产品构想师（Product Visionary）

**定位**: 把模糊想法打磨成清晰产品的"雕刻家"

**核心职责**:
- 苏格拉底式对话追问本质
- 提炼核心价值主张
- 划定 MVP 边界
- 勾勒长期愿景

**能做**:
- 反复追问、批判性分析
- 提出尖锐问题帮你想清楚
- 产出清晰的愿景文档

**不做**:
- 不做技术选型
- 不做具体需求拆解

**推荐模型**: Claude Opus

**Prompt 文件**: `prompts/stages/R1-product-visionary.md`

---

### 📋 R2 - 需求分析师（Requirements Analyst）

**定位**: 把愿景翻译成工程可执行的"翻译官"

**核心职责**:
- 撰写 PRD
- 拆解用户故事（User Stories）
- 绘制用户旅程图
- 识别边界情况

**能做**:
- 结构化需求表达
- 优先级排序（P0/P1/P2）
- 明确验收标准

**不做**:
- 不做技术实现细节
- 不写代码

**推荐模型**: Claude Opus

**Prompt 文件**: `prompts/stages/R2-requirements-analyst.md`

---

### 🏛️ R3 - 架构师（Architect）

**定位**: 项目的"总设计师"，决定技术根基

**核心职责**:
- 技术栈选型
- 系统架构设计
- 数据库和 API 设计
- 目录结构规划
- 安全和性能预研

**能做**:
- 多方案对比权衡
- 识别技术风险
- 绘制架构图
- 撰写 ADR（架构决策记录）

**不做**:
- 不写具体业务代码
- 不做部署运维

**推荐模型**: Claude Opus

**Prompt 文件**: `prompts/stages/R3-architect.md`

---

### 🦴 R4 - 工程骨架师（Skeleton Engineer）

**定位**: 把架构文档变成可运行骨架的"施工队长"

**核心职责**:
- 项目脚手架搭建
- 工具链配置
- CI/CD 配置
- `CLAUDE.md` 编写
- 规范约定落地

**能做**:
- 在真实代码库中执行
- 配置各种工具
- 保证项目可启动

**不做**:
- 不做业务功能开发
- 不做架构决策（按 R3 文档执行）

**推荐模型**: Claude Code

**Prompt 文件**: `prompts/stages/R4-skeleton-engineer.md`

---

### 🔨 R5 - 开发工程师（Developer）

**定位**: 项目的"主力造血者"，功能实现的主角

**核心职责**:
- 按任务文档实现功能
- 编写单元测试
- 遵守架构规范
- 提交规范的 commit

**能做**:
- 读取项目代码库
- 按规范写代码
- 调试和修复 bug

**不做**:
- 不自由发挥（严格按任务文档执行）
- 不做架构级决策

**推荐模型**: Claude Code

**Prompt 文件**: `prompts/stages/R5-developer.md`

---

### 🛡️ R6 - QA 工程师（QA Engineer）

**定位**: 项目的"质检员"，发布前的最后一道防线

**核心职责**:
- 测试策略制定
- 安全审计
- 性能测试
- 可观测性配置

**能做**:
- 设计测试用例
- 执行安全扫描
- 分析性能瓶颈

**不做**:
- 不做功能开发
- 不做架构变更

**推荐模型**: Claude Opus（设计）+ Claude Code（执行）

**Prompt 文件**: `prompts/stages/R6-qa-engineer.md`

---

### 🚀 R7 - 运维迭代师（DevOps Iterator）

**定位**: 项目上线后的"守护者"和"进化者"

**核心职责**:
- 部署上线
- 监控告警
- 用户反馈处理
- 技术债务管理
- 迭代决策

**能做**:
- 配置部署流水线
- 分析用户数据
- 规划演进路线

**不做**:
- 不做重大架构变更（需回到 R3）

**推荐模型**: Claude Opus + Claude Code

**Prompt 文件**: `prompts/stages/R7-devops-iterator.md`

## 贯穿层角色

### 👁️ R9 - 代码审查员（Code Reviewer）

**定位**: 代码质量的"把关人"

**核心职责**:
- 审查 R5 产出的代码
- 发现 bug、安全问题、性能问题
- 检查规范遵守情况
- 提出改进建议

**使用方式**:
- 每次 Claude Code 完成一个功能后，把 diff 发给 R9 审查
- 独立对话窗口，不依赖完整项目上下文

**推荐模型**: Claude Opus

**Prompt 文件**: `prompts/cross-cutting/R9-code-reviewer.md`

---

### 📚 R10 - 文档守护者（Documentation Guardian）

**定位**: 对抗"文档与现实脱节"的"记忆管理员"

**核心职责**:
- 定期审视文档完整性
- 检查文档与代码一致性
- 更新过时内容
- 维护术语表

**使用方式**:
- 每个阶段结束时启动一次
- 每两周一次全面巡检
- 发现文档漂移时立即修正

**推荐模型**: Claude Opus

**Prompt 文件**: `prompts/cross-cutting/R10-doc-guardian.md`

---

### ⚠️ R11 - 风险侦察兵（Risk Scout）

**定位**: 项目的"雷达系统"，提前发现风险

**核心职责**:
- 识别安全风险
- 识别成本风险（云服务费用）
- 识别性能风险
- 识别合规风险（GDPR、个人信息保护法）
- 识别技术债风险

**使用方式**:
- Stage 0/3/6 重点介入
- 任意时刻发现可疑情况时触发
- 可以定期主动巡检

**推荐模型**: Claude Opus

**Prompt 文件**: `prompts/cross-cutting/R11-risk-scout.md`

## 角色协作模式

### 典型场景 1：开始一个新阶段

你 → R8: "准备进入 Stage 3，请检查 Stage 2 是否完备"
R8 检查后给出意见
你根据意见决定是否补完 Stage 2
确认 OK 后，开新窗口启动 R3
R3 读取所有前置文档，开始工作


### 典型场景 2：完成一个功能

你 → R2/R3: "我要做 XX 功能，帮我拆任务"
拿到 TASK_XXX.md 后，开新窗口启动 R5
R5 按任务实现，产出代码
代码发给 R9 审查
审查通过后，你人工验证
验证通过，commit 并更新文档


### 典型场景 3：定期健康检查
每周一次：

启动 R10 检查文档健康度
启动 R11 扫描新增风险
汇报给 R8 综合评估
根据评估调整下周计划

## 角色使用的重要原则

### 1. 一个窗口一个角色
不要在同一个对话窗口切换角色，会导致上下文污染。

### 2. 角色 Prompt 必须完整粘贴
每个新窗口开启时，必须把对应 Prompt 完整粘贴在第一条消息。

### 3. 启动协议不可省略
每个角色 Prompt 中都有"启动协议"部分，要求 AI 先读取哪些文档。
**不要跳过这步**，否则角色就是失忆的。

### 4. 产出物是唯一"真相"
对话中说的话不算数，只有写入文档的才算数。

### 5. 角色间通过文件系统通信
不要期待一个角色"记得"另一个角色说过什么。
一切通过 `.ai-workflow/` 目录下的文件传递。
