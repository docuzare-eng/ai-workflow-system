# 最佳实践 (Best Practices)

> **读者**: 已经用过几次这套系统，想进一步提升效率和质量的人  
> **目标**: 把"会用"升级到"用得好"  
> **前置阅读**: `GETTING_STARTED.md`

---

## 一、文档写作的最佳实践

### 1.1 写文档的黄金原则

#### 原则 1: 为"未来的陌生人"写

你今天写的文档，读者可能是：
- 3 个月后失忆的自己
- 新加入的合作者
- 一个新启动的 AI 角色

**不要假设读者知道背景**。重要的 context 都写进去。

#### 原则 2: 可验证 > 可美化

**差**：
```
"系统应具备良好的性能。"
```

**好**：
```
"首屏 LCP < 2.5s，API P95 < 500ms（详见 CONSTRAINTS.md）"
```

每一条要求都要**可测试**。模糊表达是"走过场"。

#### 原则 3: 记录"为什么"比记录"是什么"更重要

**"是什么"**是代码就能告诉你的（读代码就知道用 PostgreSQL）。
**"为什么"**只有文档能告诉你（为什么选 PostgreSQL 而不是 MongoDB）。

重点记：
- 决策的背景
- 考虑过的选项
- 选择的理由
- 放弃的理由

### 1.2 `PROJECT_CONSTITUTION.md` 写好的技巧

#### 愿景陈述的 3 个特征

好愿景应该有：
- 🎯 **具体**：能看到画面
- 💪 **有野心**：让人想努力
- 📏 **可验证**：能判断是否达成

**差例**:
> "打造一个用户喜爱的产品"

**好例**:
> "2026 年底，本产品在 [细分领域] 的独立开发者中拥有 1000 位月活用户，其中 30% 愿意付费订阅。"

#### "不做什么"比"做什么"更难但更重要

清单里要有**明确的"排除项"**：

```markdown
## 明确不做

- ❌ 不做企业版（理由：运营成本高、不匹配独立开发者定位）
- ❌ 不做移动端原生 App（理由：MVP 阶段专注 Web）
- ❌ 不做 AI 图像生成（理由：与核心无关，分散注意力）
```

**没有"不做什么"** = 所有事都可能做 = 失去聚焦。

### 1.3 `CONSTRAINTS.md` 写好的技巧

#### 用 3 级分类

```markdown
## 🔴 硬性约束（不可违反）
- TypeScript 严格模式
- 参数化 SQL
- HTTPS only

## 🟡 强偏好（有强力理由才违反）
- 使用 Tailwind 而非自定义 CSS
- 使用 pnpm 而非 npm

## 🟢 软建议（倾向于遵循）
- 函数 < 50 行
- 文件 < 300 行
```

**不要把所有约束放在同一级**。区分让 AI 和人都能判断什么能妥协。

#### 约束要"有理由"

每条约束至少一句话说明理由：

**差**:
```
- 不使用 MongoDB
```

**好**:
```
- 不使用 MongoDB（理由：项目需要强事务支持，MongoDB 的事务能力不足以满足订单场景）
```

这样三个月后你或新来的人才知道"能不能破例"。

### 1.4 `DECISIONS.md` 写好的技巧

#### 写 ADR 的触发条件

不是每件事都要写 ADR。**真正要写的是**：
- ✅ 影响架构的选择（选 A 框架还是 B）
- ✅ 会让后续无法轻易改变的决策
- ✅ 多人/多 AI 都需要知道的约定
- ✅ 未来可能被质疑的决策（提前把理由记下来）

**不必写 ADR 的**：
- ❌ 临时的、局部的代码实现
- ❌ 显而易见的选择
- ❌ 已经被现有 ADR 覆盖的

#### ADR 的"后果" 部分很重要

很多人写 ADR 只写"决策 + 理由"，忘了写"后果"。

**后果要写**：
- ✅ 这个决策带来什么好处
- ⚠️ 付出什么代价
- 🔄 什么条件下需要重新评估

这样未来回看时知道"这个决策还是不是对的"。

## 二、AI 角色使用的最佳实践

### 2.1 粘贴 Prompt 的技巧

#### 第一条消息只粘贴 Prompt 本身

**不要**同时贴 Prompt + 项目信息 + 要求。让 AI 先"进入角色"，再告诉它要做什么。

#### 等 AI 自我介绍后再开始任务

好的启动流程：

```
你：[粘贴 R0 Prompt]
AI：✅ 我是 R0 可行性分析师... 请告诉我你的项目...
你：[这里描述项目]
AI：[开始工作]
```

**不要**跳过 AI 的启动自述。那个过程帮它建立角色感。

### 2.2 加载上下文的技巧

#### 按角色需要**精准**加载

每个角色的 Prompt 都列出了"必读"和"按需"文件。**精准粘贴**对应文件。

**不要**一股脑把所有文档都贴给它（会冲淡注意力，还浪费 token）。

#### 大文档分批粘贴

如果某个文档很长（> 5000 字），分批粘贴：

```
你：第一部分是 [...]
AI：好的，已记下。
你：第二部分是 [...]
AI：好的，已记下。
你：现在开始工作吧。
```

这样 AI 处理更扎实。

### 2.3 让 AI 主动"对齐"的技巧

#### 让它复述理解

每次给 AI 一个较复杂的任务后：

```
你：你开始前，先用你的话复述一遍你理解的任务和关键约束。
```

AI 复述时会暴露它的理解偏差。你及时纠正就能避免跑偏。

#### 让它列出计划

对大任务：

```
你：在开始写代码前，先列一个 5-10 步的计划，让我确认。
```

计划里能看出它是否真的理解了。你批准后它再执行。

### 2.4 长对话的管理技巧

#### 适时"蒸馏"长对话

对话超过 50 条消息时，可能出现：
- 上下文被冲淡
- 早期的约束被遗忘
- AI 开始"飘"

**解决**: 让 AI 做一次**总结**：

```
你：请总结一下我们这次对话做了什么、关键决策、
    当前状态、下一步计划。按结构化格式输出。
```

然后**新开一个对话窗口**，把这份总结作为启动上下文。

#### 重要决策立即入档

AI 有时说出非常好的洞见，但对话翻页后就难找回。

**立即**把有价值的决策记录到 `DECISIONS.md`，或 `OPEN_QUESTIONS.md`。

### 2.5 不同角色协作的技巧

#### R8 是总指挥，不是打工仔

有些新手把什么事都问 R8。**R8 的价值是全局视角**，不是什么都能做。

**问 R8**:
- 阶段间冲突怎么解决？
- 这个决策会不会破坏全局？
- 节奏对吗？
- 健康度如何？

**不要问 R8**:
- 帮我写这段代码（→ 找 R5）
- 帮我设计数据库（→ 找 R3）
- 这段代码有 bug（→ 找 R9 + R5）

#### R9 不要"凑合"

R9 代码审查要严格。有时你赶时间，想说"这个小问题下次再修"。

**原则**：
- 🔴 安全/功能/架构类问题 → **必须现在修**
- 🟡 代码质量问题 → 可以"下次修"，但**必须记到 TECH_DEBT.md**
- 🟢 风格/偏好问题 → 视情况

"下次修"但不记录 = 永远不会修 = 技术债埋雷。

#### R10 要定期，不要"想起来"

R10 文档守护者最大的价值是**定期**。

**建议节奏**：
- 每个阶段结束时一次
- Stage 5 期间每两周一次
- 发现不对劲时临时一次

如果每次都"想起来"才做，基本等于不做。

#### R11 的输出要"分级处理"

R11 发现 10 个风险，你不可能都立刻处理。它会给出"可能性 × 影响"的打分。

**处理策略**：
- 🔴 高 × 高 → 立刻处理
- 🟡 高 × 中 或 中 × 高 → 本周处理
- 🟢 其他 → 登记并定期重评估

不要被大量风险吓倒，要**有优先级**。

## 三、节奏管理的最佳实践

### 3.1 各阶段的"够了"信号

每个阶段完美没有尽头。**够了**的信号：

#### Stage 0（可行性）"够了"
- ✅ 能清晰回答"为什么做"、"为谁做"、"值得做"
- ✅ 主要风险都识别了
- ✅ R8 审查通过

不够：你还在犹豫"要不要换方向"。

#### Stage 1（愿景）"够了"
- ✅ 能用一句话说清楚产品
- ✅ MVP 功能清单 ≤ 5 个
- ✅ "不做什么"清单明确

不够：你觉得"好像还可以加点什么"。

#### Stage 2（需求）"够了"
- ✅ P0 故事工程师看了能独立实现
- ✅ 边界情况都考虑了
- ✅ 没有 XL 级故事

不够：发现需求模糊到工程师还要回来问一堆。

#### Stage 3（架构）"够了"
- ✅ 能画出完整架构图
- ✅ 关键决策都有 ADR
- ✅ 核心用例都有架构应对

不够：还在犹豫"用 A 技术还是 B 技术"。

### 3.2 避免"完美主义陷阱"

每个阶段都可能花**无限时间**做到"完美"。这是陷阱。

**铁律**：
- 每阶段**超过预估 2 倍时间**还没完成 → 肯定有问题
- 问题可能是：完美主义 / 范围蔓延 / 遇到真障碍

找 R8 协调官分析，判断是哪种情况。

### 3.3 "回退" vs "缝合"的判断

做到 Stage 5 发现 Stage 3 的架构有问题。该怎么办？

**小问题（缝合）**：
- 影响局部
- 不会引发连锁
- 修复成本低
- → 在当前阶段打补丁 + 记 ADR

**大问题（回退）**：
- 影响多个模块
- 会持续累积
- 不修后面越做越痛
- → 咬牙回退到 Stage 3 修

**判断标准**: 修复成本 vs 不修后续总成本。

找 R8 评估。

### 3.4 Stage 5 的节奏窍门

Stage 5 是最长的阶段。保持节奏的关键：

#### "一次只做一个 FEAT"

避免同时做 3 个功能。AI 和人都容易混乱。

**完整流程**:
```
拿到 FEAT-XXX → 开始 → 开发 → 自测 → 提 PR → R9 审 → 修 → 合并 → 标记完成 → 下一个
```

#### 每天结束时"归零"

下班前：
- ✅ 把未完成的代码 commit（即使是 WIP）
- ✅ 在 FEAT-XXX.md 里记下"今天做到哪"
- ✅ 告诉 R8 当前状态

第二天来：
- ✅ 看 FEAT-XXX.md 回忆上下文
- ✅ 从明确的下一步开始

不要"让工作悬着"，否则第二天恢复状态要花很久。

#### 每周一次 R8 + R10 "体检"

每周五或周一花 30 分钟：
- R8 评估本周进度和健康度
- R10 检查文档有没有漂移
- 两者给出下周建议

这 30 分钟防止小问题累积成大问题。

## 四、协作的最佳实践

### 4.1 和 AI 的协作

#### 对 AI 要"直接但不粗鲁"

**好**:
> "我觉得你这个方案有问题，因为 X 会导致 Y。能再考虑一下吗？"

**差**（过于客气）:
> "不好意思打扰你，你这个方案可能不是最好的..."

**差**（粗鲁）:
> "你这完全错了"

AI 不需要你照顾情绪，但也不应该被当出气筒。**像同事一样说话**。

#### 给 AI 反馈时要具体

**差**: "这个不对"  
**好**: "这里不对：[具体位置]。原因是：[...]。建议：[...]"

具体的反馈让 AI 能迅速修正。模糊的反馈导致它瞎猜。

#### 对 AI 要有"边界感"

AI 会试图帮到你**任何事**，包括你不该问它的。

**不要问 AI**:
- 法律建议（找律师）
- 医疗建议（找医生）
- 个人情感决定（找你自己）
- 具体客户的商业秘密

### 4.2 如果是多人协作

#### 建立"主角色窗口"规则

约定谁负责哪个角色：
- Alice 负责 R1/R2（产品/需求）
- Bob 负责 R3/R4（架构/骨架）
- 大家都可以启动 R5（开发）
- R8/R9/R10/R11 按需

避免两个人同时和同一个角色窗口"拉扯"。

#### 用 Git 同步状态

所有人都从 GitHub 拿最新文档。**不要**自己留一份本地不同步。

如果有冲突（比如两人都改了 DECISIONS.md），正常 Git 解决冲突即可。

#### 每周 Sync 会议（可选）

多人时建议有 Sync 节奏：
- R8 主持（人来扮演）
- 过 WORKFLOW_STATE.md
- 讨论 OPEN_QUESTIONS.md
- 下周规划

## 五、长期维护的最佳实践

### 5.1 文档"保鲜"的窍门

文档最怕的不是"写得不好"，而是**"和现实脱节"**。

#### 铁律 1: 代码和文档绑定变更

**改代码**同时**改文档**，否则拒绝 PR。

例：改了 API → 必须同步改 `API_DESIGN.md`。

R9 审查代码时，**也审查文档更新**。

#### 铁律 2: 周期性巡检

R10 文档守护者**定期**跑：
- 每阶段结束
- 每 2 周一次
- 感觉不对劲时

#### 铁律 3: 接受"文档债"的概念

有时赶进度来不及更文档。这是**文档债**，和技术债一样：
- 允许有
- 但必须记录
- 必须定期偿还

在 `TECH_DEBT.md` 里专门建一个"文档债"分类。

### 5.2 决策回溯的智慧

3 个月后你可能觉得"当初这个决策不对"。这很正常。

**处理流程**：
1. 先读 `DECISIONS.md` 里这个决策的**当时背景**
2. 判断：是**当时就错**了，还是**情况变化**了
3. 写一个**新的 ADR**，标记"取代 ADR-XXX"
4. 执行新决策
5. 保留旧 ADR（不删除）

**不要**偷偷改旧 ADR。**不要**说"我们从来没这么决定过"。

这种"诚实的修正"是健康项目的标志。

### 5.3 知识沉淀

每个阶段结束时，在 `DAILY_LOG.md` 写一段总结：
- 本阶段做了什么
- 学到了什么
- 踩了哪些坑
- 如果重来会怎么做

**3 个月后这些笔记就是宝藏**。

### 5.4 对于新加入的人

如果有新人加入（人或 AI），给他们**30 分钟入门包**：

```
请依序读：
1. README.md（5 分钟）- 项目简介
2. PROJECT_CONSTITUTION.md（5 分钟）- 项目宪法
3. VISION.md（5 分钟）- 产品愿景
4. WORKFLOW_STATE.md（3 分钟）- 当前状态
5. ARCHITECTURE.md 的概览部分（5 分钟）
6. CLAUDE.md（5 分钟）- 开发速查

其他文档按需查。
```

30 分钟后，他们应该能**理解项目**并**开始贡献**。

## 六、反模式（什么是错的）

### ❌ 反模式 1: "只用 R5 开发"

有人觉得："角色太多了，我就用 R5 写代码不行吗？"

**问题**:
- 没做 Stage 0-4 → 做了错误的东西
- 没用 R9 → 代码质量不可控
- 没用 R10 → 文档漂移
- 没用 R8 → 节奏失控

**后果**：三个月后代码成山，但做的都不是用户要的。

**正确**: 按流程走。最少也要做 Stage 1 + Stage 2 + Stage 3 + 用 R5 + 用 R9。

### ❌ 反模式 2: "所有角色共享一个窗口"

有人觉得："开这么多对话窗口太乱了，我把所有提问都放一个窗口。"

**问题**:
- AI 混淆不同角色
- 上下文冲突
- 质量下降

**正确**: 一个角色一个窗口。R8 保持长期，其他角色一次性用完关闭。

### ❌ 反模式 3: "文档只写不看"

有人认真填了 `VISION.md`、`REQUIREMENTS.md`，然后……就没然后了。

**问题**: 文档的价值在于**被引用**。不看的文档 = 不存在的文档。

**正确**: 
- 每开新对话就给 AI 贴上相关文档
- 自己决策时 Ctrl+F 搜一下 DECISIONS.md
- 每周至少扫一眼 WORKFLOW_STATE.md

### ❌ 反模式 4: "只填空不思考"

有人把模板当作填空题，速速填完每个 `[...]`。

**问题**: 模板只是**脚手架**。真正的价值在**每个字段的思考过程**。

**正确**: 填每个字段前问自己"这个真的是对的吗？有依据吗？"

### ❌ 反模式 5: "AI 说啥都对"

有人完全信任 AI，不加判断。

**问题**: AI 会犯错，会胡编，会"讨好"你的想法。

**正确**: 
- 对重大决策，至少从两个角度（两个角色）审视
- 你是最终决策者，不是 AI
- 有疑问问 R8 或 R10

### ❌ 反模式 6: "完美主义不发布"

有人做到 Stage 6，还在"再改改再改改"。

**问题**: 好产品是**用出来**的，不是**写出来**的。迭代比完美更重要。

**正确**: MVP 达到"够用"就发布。持续迭代。

### ❌ 反模式 7: "上线后不再维护"

有人上线了 = 项目结束 = 不再更新文档。

**问题**: 产品持续在变，文档不更 = 6 个月后团队没人懂。

**正确**: 运维期也定期用 R10 + R11 巡检。每次重大变更都同步文档。

## 🎯 终极原则

用三句话总结所有最佳实践：

1. **尊重流程**：流程是经验的结晶，偷懒代价高。
2. **尊重文档**：文档是项目的大脑，忽视它等于自废武功。
3. **尊重节奏**：好项目靠节奏，不靠爆发。
