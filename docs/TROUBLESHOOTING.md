# 故障排查 (Troubleshooting)

> **读者**: 使用中遇到问题的任何人  
> **目标**: 快速找到问题和解决方案  
> **使用方式**: 按症状/情境查找，不用按顺序读

---

## 📖 使用说明

### 查找方式

- **按症状**: 用 Ctrl+F 搜关键词（如"跑偏"、"忘记"、"矛盾"）
- **按阶段**: 看第六、七部分按阶段组织的问题
- **按角色**: 看第八部分按角色组织的问题

### 问题格式

每个问题按以下格式：

```
## 🔍 症状描述
（你遇到的现象）

### 💡 可能原因
1. 原因 A
2. 原因 B

### ✅ 解决方案
具体步骤

### 🛡️ 预防措施
如何避免再次发生
```

## 一、AI 相关问题

### 🔍 AI 不按角色 Prompt 的要求工作

**场景**: 你粘贴了 R3 架构师的 Prompt，但它开始写代码（那是 R5 的事）。

**可能原因**:
1. Prompt 没粘贴完整（被截断）
2. 你粘贴 Prompt 时同时说了"帮我写点代码"，AI 按后面的指令工作
3. 对话已经很长，早期的 Prompt 被淡化

**✅ 解决方案**:

**方案 A**（推荐）: 提醒它

```
提醒：你现在是 R3 架构师。请严格按 Prompt 的工作流程，
只做架构设计，不写具体代码。现在请回到架构设计工作。
```

**方案 B**: 开新窗口

如果提醒无效或对话已很长，开一个**全新对话**，重新粘贴完整 Prompt。

**🛡️ 预防**:
- 粘贴 Prompt 时不要同时问别的
- 让 AI 先"自我介绍"后再给任务
- 对话长了（> 30 条消息）考虑开新窗口

---

### 🔍 AI 忘记了之前的决策

**场景**: 项目早期决定用 PostgreSQL，但第 3 周 AI 开始说"我们用 MongoDB 吧"。

**可能原因**:
1. 当前窗口没加载 `DECISIONS.md`
2. 决策没有及时入档 → AI 从训练数据猜测
3. 长对话中早期的约束被"淡化"

**✅ 解决方案**:

1. **立即修正**:
```
注意：我们在 ADR-012 已经决定使用 PostgreSQL（不用 MongoDB），
理由是需要强事务支持。请继续按 PostgreSQL 设计。
```

2. **检查 `DECISIONS.md`**: 如果这个决策确实没记录，**立即补上**
3. **下次启动 AI 时，务必贴 `DECISIONS.md`**

**🛡️ 预防**:
- 每个重要决策**立即**进 `DECISIONS.md`
- 每次启动新对话都贴相关决策
- 定期让 R10 做一致性检查

---

### 🔍 AI "跑偏"或给出离谱建议

**场景**: R5 开发工程师突然建议你"重构整个架构"或"换一个框架"。

**可能原因**:
1. 缺少必要的上下文（它不知道约束）
2. 被某段错误信息带歪
3. 它的模型幻觉

**✅ 解决方案**:

**第一步: 立即打断**
```
停一下。我觉得这个建议有问题。
在继续前，请先回答：
1. 这个建议是否违反 CONSTRAINTS.md？
2. 当前任务真的需要这个改动吗？
3. 如果不做这个，会发生什么？
```

让它自己审视一遍。大部分"跑偏"在这一步就回来了。

**第二步: 如果还坚持**
- 启动 R8 或 R10 做交叉审视
- 你作为决策者拍板
- 记录到 `DECISIONS.md`（说"评估了但不采纳"）

**🛡️ 预防**:
- 给 AI 明确的任务边界
- 及时刷新它对约束的认知
- 重大建议**绝不**立即采纳

---

### 🔍 AI 编造不存在的东西（幻觉）

**场景**: AI 引用了一个**不存在**的 ADR 编号，或说"我们之前约定了 X"但实际从没约定过。

**可能原因**: LLM 的幻觉问题，无法完全避免。

**✅ 解决方案**:

**立即验证**:
```
你刚才说"ADR-XXX"和"我们约定过 X"。
请你具体指出这在哪个文档的哪一章节？
如果找不到，请承认你可能记错了。
```

好的 AI 会承认。差的 AI 会继续编——这时**切换模型**或**开新窗口**。

**🛡️ 预防**:
- 关键决策都**要求 AI 引用具体文档位置**
- 对"我记得"类表述保持怀疑
- 重要结论自己 Ctrl+F 验证

---

### 🔍 Claude 显示"上下文太长了"或响应变慢

**场景**: 对话进行到一定程度，Claude 开始响应慢、忽略细节、甚至报错。

**可能原因**: 对话累积的 token 接近上下文窗口上限。

**✅ 解决方案**:

**方案 A（推荐）: 蒸馏 + 新窗口**

1. 让当前窗口做一次总结：
```
请总结我们这次对话：
- 做了什么
- 关键决策
- 当前状态  
- 未解决问题
- 下一步

按结构化格式输出。
```

2. 保存这份总结
3. 开新窗口，把总结作为"当前项目状态"提供给它

**方案 B: 精简对话（不推荐）**

让 AI 忘记前期细节，只保留关键信息。但不够可靠。

**🛡️ 预防**:
- 长任务分多个窗口完成
- 每完成一个阶段做一次"蒸馏"
- 关键信息及时入档（不靠对话记忆）

## 二、文档相关问题

### 🔍 文档和代码对不上

**场景**: `API_DESIGN.md` 说有 `/api/users` 端点，但代码里找不到。

**可能原因**:
1. 文档写了但代码没实现
2. 代码改了但文档没同步
3. 两者都对，是第三个文件（路由配置）错了

**✅ 解决方案**:

1. **启动 R10 文档守护者**做**同步巡检**（type D）
2. R10 会列出所有不一致
3. 对每个不一致决定：
   - 文档是对的 → 补代码
   - 代码是对的 → 改文档
   - 两者都对一部分 → 重新对齐

**🛡️ 预防**:
- R9 审查代码时**同时审查文档更新**
- 每 2 周让 R10 做一次同步巡检
- 建立"代码+文档一起提交"的文化

---

### 🔍 某个决策不知道在哪里

**场景**: 你记得讨论过"要不要用 Redis"，但找不到结论在哪。

**✅ 解决方案**:

**按顺序查找**:
1. `DECISIONS.md` Ctrl+F 关键词
2. `OPEN_QUESTIONS.md` Ctrl+F（可能还没决策）
3. `CONTEXT_HANDOFF.md`（可能在交接笔记里）
4. `DAILY_LOG.md`（按日期翻）

**如果都找不到**:
→ 说明这个决策**没有被正式记录**  
→ **现在就补一个 ADR**：

```
补记 ADR-XXX（后补于 YYYY-MM-DD）

背景: 我们在 [大概日期] 讨论过 [决策]
决策: [如果记得是什么]
理由: [如果记得]
备注: 此 ADR 为事后补记，因此可能有细节不准确
```

**🛡️ 预防**:
- 讨论时就记笔记
- 结论出来当场进文档
- 不依赖记忆

---

### 🔍 多人改同一份文档有冲突

**场景**: 你和合作者都在改 `DECISIONS.md`，Git 合并冲突。

**✅ 解决方案**:

**预防性做法**:
- 大段修改前说一声
- 频繁 pull 同步

**已经冲突时**:
1. 不要手动解决——容易出错
2. 两人一起看冲突
3. **两个变更都保留**（都是有意义的）
4. 协商先后顺序

**更好的方式**:
- 用 GitHub PR 机制
- 单独分支改
- 合并前 review

**🛡️ 预防**:
- 约定"同时只有一人改某文档"
- 用 PR 流程

## 三、流程相关问题

### 🔍 已经做到 Stage 5 了，才发现 Stage 2 的需求有问题

**场景**: 写代码写到一半，发现某个需求理解错了，或者用户真实需要不是这个。

**✅ 解决方案**:

这是**双向回流**机制要处理的情况。

1. **启动 R8 协调官**，说明情况
2. R8 判断严重性：

**轻微**:
- 单个用户故事理解偏差
- 修复不影响其他

→ 在 Stage 5 内 patch + 在 `DECISIONS.md` 记录

**中度**:
- 多个故事都有偏差
- 但架构还能承受

→ 回到 Stage 2 更新需求，但架构不变 + Stage 5 继续

**重度**:
- 需求整体错了
- 架构也需要重新设计

→ 咬牙回到 Stage 2（或 Stage 1），**坦然接受代价**

3. **无论哪种**，都要记录这次"回流"：
   - 为什么之前没发现
   - 这次学到什么
   - 下次如何避免

**🛡️ 预防**:
- Stage 2 结束时做更严格的 R8 审查
- 早期用"用户测试"验证需求真实性
- 不赶时间跳过 Stage 0 的可行性验证

---

### 🔍 某个 Stage 做了很久也做不完

**场景**: Stage 3 架构设计已经做了 3 周还没结束。

**可能原因**:
1. **完美主义**: 想做到"完美架构"
2. **范围蔓延**: 架构越设计越大
3. **技术难点**: 遇到真的难搞的问题
4. **决策恐惧**: 不敢拍板选 A 还是 B

**✅ 解决方案**:

**第一步: 找 R8 诊断**

```
我在 Stage 3 已经 3 周了。请帮我诊断：
1. 我遇到的是什么类型的问题？
2. 哪些是过度设计？
3. 哪些是真的卡点？
4. 我应该如何推进？
```

**第二步: 按问题类型处理**

**完美主义**: R8 会强制你"砍成 80 分"交付
**范围蔓延**: 回到 `VISION.md` 收敛范围
**技术难点**: 做小 POC 验证 + 决策
**决策恐惧**: 强制二选一，记 ADR，不后悔

**第三步: 如果真需要更多时间**

- 评估是否值得延期
- 如果不值得，砍范围保时间
- 如果值得，和自己坦诚"项目更大了"

**🛡️ 预防**:
- 每阶段前预估时长
- 每阶段中期让 R8 做一次中期检查
- 不要追求完美，追求"够好"

---

### 🔍 项目推进太慢，感觉一直没进展

**场景**: 做了一个月，感觉没做出什么东西。

**可能原因**:
1. 都在"前期阶段"（Stage 0-3 都是规划，不是出东西）
2. 真的卡住了
3. 节奏太慢（每天只工作 1 小时）
4. 范围过大

**✅ 解决方案**:

**启动 R8 做健康度巡检**:
```
帮我看看我的项目进展是否正常。
我已经做了 X 周，当前在 Stage Y。
```

R8 会告诉你：
- 进度是否正常（不同阶段合理时长不同）
- 是否有隐性问题
- 建议调整节奏或范围

**特殊处理**:

- **前期阶段慢**: 正常，但要有产出（文档是产出）
- **Stage 5 慢**: 拆得太大了，单个 FEAT 应该 1 周内完成
- **一直卡同一步**: 真有障碍，需要专门解决

**🛡️ 预防**:
- 每周至少进度小结
- 设定"合理时长"
- 超过 1.5 倍预估就警觉

---

### 🔍 门禁清单总是通不过

**场景**: 每次找 R8 做门禁审查都说"未通过"。

**可能原因**:
1. 你确实没做到标准
2. 你对标准理解偏差
3. R8 太严格（少见）

**✅ 解决方案**:

**第一步: 仔细看未通过的具体项**

R8 会列出哪些 🔴 必须项未打钩。**针对性解决**，不要笼统。

**第二步: 判断是"真没做"还是"做了但没写"**

很多时候你做了工作但没更新文档，也会"没通过"。

**第三步: 如果你觉得标准过严**

找 R8 讨论:
```
我觉得 gate-XX 清单的第 N 项对我的项目可能不适用，
因为 [理由]。能否豁免？
```

R8 会判断是否合理豁免。豁免**要记录**。

**🛡️ 预防**:
- 在每阶段**进行中**就对照清单自检
- 不要"感觉做完了"才去审查
- 清单是引导，不是障碍

## 四、角色相关问题

### 🔍 R8 给的建议总是"太保守"或"太激进"

**场景**: R8 要么什么都说"要谨慎"，要么什么都说"可以推进"。

**可能原因**:
1. 它没加载足够上下文（不知道项目情况）
2. 对话中你的情绪影响了它的基调

**✅ 解决方案**:

**刷新上下文**:
```
在继续之前，请让我重新给你贴核心文档：
- PROJECT_CONSTITUTION.md
- CONSTRAINTS.md
- WORKFLOW_STATE.md
- 最近的 DECISIONS.md

然后以这些为准，重新评估我的项目健康度。
```

**保持中立语气**: 你问问题时尽量中立，不要"诱导性"问法。

**差**: "我觉得这个风险不大，你说对吗？"  
**好**: "这个风险你评估一下是 🟢🟡🔴 哪级？"

---

### 🔍 R9 审查太严，每次都说一堆问题

**场景**: 提交一段代码，R9 列出 10+ 个问题，打击士气。

**可能原因**:
1. 代码确实有很多问题（那就是好事，被抓出来了）
2. R9 没区分优先级
3. 你的代码规范确实不够严格，R9 在"好心教育"

**✅ 解决方案**:

**让 R9 分优先级**:
```
谢谢反馈。请帮我把这些问题按优先级分组：
- 🔴 必须改才能合并
- 🟡 强烈建议改
- 🟢 可以延后

我会按优先级处理。
```

**对 🟢 问题批量处理**:
- 能用 linter 自动修的 → 让 linter 跑一遍
- 小细节统一 → 建立规范文档 + 一次性修

**🛡️ 预防**:
- 提交前**自查**常见问题
- 用 pre-commit hook 过 linter
- 熟悉 `CONSTRAINTS.md` 的要求

---

### 🔍 R10 每次都发现文档问题，感觉永远改不完

**场景**: 每次跑 R10 都列出一堆不一致，打击继续的动力。

**✅ 解决方案**:

**认清真相**: 文档漂移是**持续的战斗**，不是一次性修复。

**分级处理**:
- 🔴 严重一致性问题（如决策矛盾）→ 立即修
- 🟡 重要但不紧急 → 本周修
- 🟢 小问题 → 批量定期修

**建立"文档修复 Sprint"**:
每个月留半天专门修文档。

**🛡️ 预防**:
- R9 审查代码时**同时审查文档更新**
- 代码+文档**一起提交**
- 不积累文档债

## 五、环境/工具相关问题

### 🔍 Codespaces 启动慢或频繁断开

**场景**: Codespaces 每次要等很久启动，或工作中突然断开。

**✅ 解决方案**:

**方案 A**: 保持常开（Codespaces 闲置 30 分钟会停）
**方案 B**: 用本地开发环境替代
**方案 C**: 升级到更大规格的 Codespaces

---

### 🔍 Git 操作卡住或失败

**✅ 解决方案**:

**常见情况**:

1. **`git push` 失败**:
```bash
# 可能远程有更新，先拉
git pull --rebase
git push
```

2. **合并冲突**:
```bash
# 手动解决文件里的 <<<<< ===== >>>>> 标记
# 保存后
git add .
git rebase --continue
```

3. **不小心提交了敏感信息**:
```bash
# 立即撤销未 push 的提交
git reset HEAD~1

# 如果已 push，更复杂（需要 force push 和重置密钥）
# 重要: 泄露的密钥必须立即轮换
```

---

### 🔍 Claude.ai 本身的问题

**场景**: Claude.ai 响应慢、报错、无法使用。

**✅ 解决方案**:

1. 检查 [Claude Status](https://status.anthropic.com/)
2. 刷新页面或重新登录
3. 如果某个对话窗口卡住，开新窗口
4. 还不行就用 API 替代（如果是 Developer）

## 六、按阶段的常见问题

### Stage 0 可行性验证

**❓ "我对项目不够了解，R0 问什么我答不上"**  
→ 正常。先做一些**桌面调研**（看竞品、看论坛讨论），再回来。

**❓ "R0 说 No-Go，我不服气"**  
→ 重新审视它的理由。如果仍觉得它偏见，找 R8 做二次审视。**最终你是决策者**，但要坦诚面对理由。

---

### Stage 1 愿景

**❓ "愿景一直很模糊，写不出具体画面"**  
→ 问自己："一年后如果成功，会发生什么具体事情？" 答案就是愿景的素材。

**❓ "MVP 功能怎么都砍不下去"**  
→ 用"如果只能保留一个呢？"的残酷选择逼自己聚焦。

---

### Stage 2 需求

**❓ "用户故事太多了，写不完"**  
→ 先写 P0 的。P1/P2 可以粗略，P3 只列标题。

**❓ "边界情况想不全"**  
→ 对每个故事问："空的怎样？错的怎样？并发怎样？权限怎样？"

---

### Stage 3 架构

**❓ "架构选择太多，选哪个焦虑"**  
→ 默认保守选择（主流、成熟、你熟悉）。除非有强理由用新技术。

**❓ "担心架构以后无法扩展"**  
→ 记住"可逆决策快做"。反正以后不对可以改，只要不是不可逆的就别焦虑。

---

### Stage 4 骨架

**❓ "骨架搭了几天还不完成"**  
→ 可能想得太完美了。**能跑起来的 80% 骨架** > **完美的 0% 骨架**。

---

### Stage 5 开发

**❓ "FEAT 太复杂，做不动"**  
→ 可能是 XL 级。**拆分**成多个 FEAT。

**❓ "代码越写越乱"**  
→ 该用 R10 做一次文档/代码一致性检查了。可能方向漂了。

**❓ "技术债越来越多焦虑"**  
→ **记录**（这就是 `TECH_DEBT.md` 的价值）。安排每月一次"还债 Sprint"。

---

### Stage 6 质量

**❓ "发现很多 bug 修不完"**

## 七、心态相关问题

### 🔍 感觉用这套系统太麻烦、想放弃

**可能原因**: 
- 开始成本高（要读很多文档、写很多文档）
- 看不到短期收益

**✅ 解决**:

**理解曲线**:
- 第 1 个项目：会觉得**很重**。但这是学习成本。
- 第 2 个项目：开始顺手，效率提升。
- 第 3 个项目：形成直觉，不用想就能用好。

**坚持建议**:
- **缩小范围**：第一个项目选 2 周能做完的
- **不追求 100%**：允许某些步骤"将就"
- **记录成功时刻**：做成一个 FEAT 就庆祝一下

---

### 🔍 觉得 AI 做得比我好，怀疑自己的价值

**场景**: AI 规划比你好、代码比你好，感觉自己只是"复制粘贴"。

**✅ 解决**:

**认清真相**: 
- AI 是**工具**，你是**决策者**
- AI 能做 1000 次 A，但选 A 还是 B 是你的判断
- AI 会给出 10 个方案，挑哪个是你的事

**你的独特价值**:
- **真实动机**: AI 不知道你为什么做这个
- **判断力**: 同样的建议，你决定是否采纳
- **坚持力**: 项目完成 = 你没放弃，而不是 AI 没放弃
- **品味**: 最终"好不好"是你说了算
