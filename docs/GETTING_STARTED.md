# 快速上手指南 (Getting Started)

> **读者**: 第一次使用 ai-workflow-system 的人（也包括半年后回来的自己）  
> **目标**: 30 分钟内从"看到这份文档"到"已经在新项目中用起来"  
> **前置要求**: 有一个 Claude.ai 账号（推荐 Pro/Team，有 Opus 模型）

---

```
一、这套系统是什么？（3 分钟读完）
```

## 一、这套系统是什么？

### 一句话

**"一套让 AI 在复杂项目开发中不迷路、不忘事、不跑偏的工作流系统。"**

### 它解决什么问题？

你用 Claude/ChatGPT 做项目时，有没有遇到过：

- 🤔 第 3 天 AI 忘了你第 1 天的决定
- 😤 对话开多了，AI 开始混淆不同项目的上下文
- 😩 每次新对话都要重复解释项目背景
- 😨 做到一半发现架构和需求对不上
- 😴 文档写了没人看，代码越写越混乱

这套系统通过 **7 阶段流程 + 12 个 AI 角色 + 分层文档记忆** 来系统性地解决这些问题。

### 它不是什么？

- ❌ **不是**一个现成的 Web 应用（它是元系统/模板）
- ❌ **不是**自动化工具（它是方法论 + 文档模板 + AI Prompt 集合）
- ❌ **不是**银弹（它要求你认真对待，才能获得收益）

### 它适合谁？

✅ **适合**:
- 独立开发者做中大型项目（> 2 周工作量）
- 小团队（1-5 人）需要结构化协作
- 任何想让 AI 辅助开发变得可控的人

❌ **不适合**:
- 做 "hello world" 级别的练手（太重）
- 简单脚本、小工具（不值得）
- 拒绝写任何文档的人（这套系统的基础就是文档）

## 二、核心概念速览

### 🎯 核心概念 1：7 个阶段

任何项目都经过这 7 个阶段（Stage 0-7）：

```
Stage 0: 可行性验证    → 要不要做？
Stage 1: 构想打磨      → 做什么？
Stage 2: 需求规格化    → 具体做什么？
Stage 3: 架构设计      → 怎么做？
Stage 4: 骨架搭建      → 搭架子
Stage 5: 功能开发      → 写代码
Stage 6: 质量保障      → 测试打磨
Stage 7: 发布迭代      → 上线维护
```

每个阶段有**准入标准**（什么时候可以开始）和**准出标准**（什么时候可以结束）。

### 🎭 核心概念 2：12 个 AI 角色

每个阶段对应一个专业角色（R0-R7），加 4 个跨阶段角色（R8-R11）：

| 编号 | 角色 | 职责 |
|------|------|------|
| R0 | 可行性分析师 | 评估要不要做 |
| R1 | 产品构想师 | 打磨产品愿景 |
| R2 | 需求分析师 | 拆解用户故事 |
| R3 | 架构师 | 设计系统架构 |
| R4 | 工程骨架师 | 搭建项目骨架 |
| R5 | 开发工程师 | 实现功能 |
| R6 | QA 工程师 | 质量保障 |
| R7 | 运维迭代官 | 发布与运营 |
| R8 | 项目协调官 | 全局把控（长期角色） |
| R9 | 代码审查员 | 代码质量把关 |
| R10 | 文档守护者 | 对抗文档漂移 |
| R11 | 风险侦察兵 | 识别项目风险 |

**关键**: 一个 Claude.ai 对话窗口 = 一个角色。切换角色 = 开新对话窗口。

### 📚 核心概念 3：分层文档记忆

AI 没有长期记忆，但**文档可以**。所以这套系统让 AI 把重要信息**写进文档**，下次通过**读文档**恢复记忆。

```
📁 .ai-workflow/        ← AI 的"外置记忆"
├── 00-core/            ← 全局基座（项目宪法、决策、约束）
├── 01-feasibility/     ← 每阶段独立目录
├── ... 
├── meta/               ← 跨阶段的交接笔记、工作流状态
└── logs/               ← 历史记录
```

### 🚦 核心概念 4：质量门禁

每个阶段结束时有"**门禁清单**"（`quality-gates/gate-XX.md`）。就像闯关游戏——不达标不能进下一关。

这防止你"觉得差不多了就往下走"，然后半道崩盘。

### 🏗️ 核心概念 5：template vs 实例

仓库的 `template/.ai-workflow/` 是**模板**——里面都是空模板（填空题）。

开新项目时，你把整个 `template/.ai-workflow/` **复制**到新项目里，然后**填空**。

## 三、开启你的第一个项目

### 总览：你要做的 8 件事

```
Step A (5 min):  准备环境
Step B (3 min):  复制模板到新项目
Step C (2 min):  初始化核心档案（宪法、约束）
Step D (5-20 min): 启动 R0 做可行性评估
Step E (根据项目而定): 通过 R8 审查，进入 Stage 1
Step F: 依次经过 Stage 1-7，用对应角色
Step G: 持续使用 R8-R11 辅助角色
Step H: 上线后迭代
```

---

### 🅰️ Step A：准备环境（5 分钟）

#### A.1 检查你有的

- [ ] **Claude.ai 账号**（推荐 Pro 或 Team，能用 Opus 模型）
- [ ] **GitHub 账号**
- [ ] **本地开发环境**（VSCode 或 Codespaces）

#### A.2 Clone 本系统

```bash
# 方式 1: 如果你想把 ai-workflow-system 作为独立工具
git clone https://github.com/[your-username]/ai-workflow-system.git

# 方式 2: 如果你只需要模板，下载 zip 解压即可
```

---

### 🅱️ Step B：为新项目复制模板（3 分钟）

#### B.1 创建新项目仓库

在 GitHub 创建一个新仓库（比如 `my-awesome-project`），clone 到本地。

#### B.2 复制 template 到新项目

**方式 A：命令行（推荐）**

```bash
# 假设 ai-workflow-system 和 my-awesome-project 都在本地
cd my-awesome-project

# 复制 .ai-workflow 目录
cp -r ../ai-workflow-system/template/.ai-workflow .

# 验证
ls -la .ai-workflow/
# 应该看到 00-core, 01-feasibility, ... 08-iteration, meta, logs

# 提交
git add .ai-workflow
git commit -m "chore: bootstrap .ai-workflow from ai-workflow-system"
git push
```

**方式 B：手动复制**

1. 从 ai-workflow-system 的 `template/.ai-workflow/` 选中所有文件
2. 复制到新项目的根目录下
3. 得到新项目的 `.ai-workflow/` 目录

> 💡 **为什么用点号开头的目录名？**  
> `.ai-workflow` 是隐藏目录。这样它不会污染你项目的主视图，但工具仍能访问。

---

### 🅲️ Step C：初始化核心档案（2 分钟）

打开 `.ai-workflow/00-core/` 下的 4 个文件，**最低限度**填写项目基本信息：

#### C.1 `PROJECT_CONSTITUTION.md`

最少填：
- 项目代号和一句话描述
- 目标用户（初步）
- 不做什么（非范围）

> ⚠️ 完整版会在 Stage 1 后由 R1 完善。现在不用填完。

#### C.2 `CONSTRAINTS.md`

最少填：
- 技术栈偏好（如果已知）
- 预算上限（如果有）
- 禁用技术（如果有）

#### C.3 `DECISIONS.md` 和 `GLOSSARY.md`

这两个先留空，项目推进过程中会自然填充。

---

### 🅳️ Step D：启动 R0 做可行性评估（5-20 分钟）

这是你第一次真正使用 AI 角色系统的时刻！

#### D.1 打开 Claude.ai

- 新建对话
- 推荐选择 **Claude Opus**（或当前最强模型）

#### D.2 找到 R0 的 Prompt

在 ai-workflow-system 中找到 `prompts/stages/R0-feasibility-analyst.md`，**全选复制**。

#### D.3 粘贴到 Claude.ai

**把 R0 的 Prompt 作为对话的第一条消息发送**。

#### D.4 按 R0 的引导走

R0 会：
1. 声明自己是"可行性分析师"
2. 引导你描述想做的项目
3. 从五个维度（市场、技术、资源、战略、个人）评估
4. 给出决策建议（Go / No-Go / Pivot）
5. 产出 `FEASIBILITY_REPORT.md` 内容

#### D.5 保存 R0 的产出

R0 给出报告后，把内容**复制**到新项目的 `.ai-workflow/01-feasibility/FEASIBILITY_REPORT.md`。

#### D.6 提交

```bash
git add .ai-workflow/01-feasibility/
git commit -m "feat(stage-0): complete feasibility analysis"
git push
```

---

### 🅴️ Step E：通过 R8 审查，决定是否推进（5 分钟）

R0 给出了报告，但还需要**R8 项目协调官**做门禁审查。

#### E.1 新开一个 Claude.ai 对话

这次是 R8 的窗口。

#### E.2 粘贴 R8 的 Prompt

找到 `prompts/cross-cutting/R8-project-coordinator.md`，全选复制，粘贴。

#### E.3 R8 会要求你提供信息

它会要：
- 项目基本信息
- 当前阶段（你说"刚完成 Stage 0"）
- 仓库 URL

#### E.4 把 R0 的报告粘给 R8

告诉 R8："我需要你做 Stage 0 的门禁审查"，然后把 `FEASIBILITY_REPORT.md` 的内容贴给它。

#### E.5 R8 会走 `gate-00-feasibility.md` 的清单

它会：
- 逐项检查
- 指出不足
- 给出最终决策：🟢 通过 / 🟡 有条件 / 🔴 未通过

#### E.6 保持 R8 窗口活着

**重要**: R8 是**长期角色**。不要关这个窗口！整个项目期间你都用同一个 R8 窗口，让它积累完整记忆。

---

### 🅵️ Step F：依次经过 Stage 1-7（按需时长）

#### F.1 每个阶段的标准流程

```
1. 从 R8 窗口问："下一步启动哪个角色？注意什么？"
2. R8 给你 briefing
3. 新开一个对话窗口
4. 粘贴对应角色的 Prompt（prompts/stages/R?-*.md）
5. 按角色引导工作
6. 产出保存到 .ai-workflow/0X-阶段名/ 目录
7. 提交 Git
8. 回到 R8 窗口做门禁审查
9. R8 通过后，写 CONTEXT_HANDOFF 给下一阶段
10. 进入下一 Stage
```

#### F.2 各阶段启动哪个角色？

| Stage | 角色 | Prompt 位置 | 主要产出 |
|-------|------|------------|---------|
| 0 | R0 | `prompts/stages/R0-feasibility-analyst.md` | `01-feasibility/*` |
| 1 | R1 | `prompts/stages/R1-product-visionary.md` | `02-vision/VISION.md` |
| 2 | R2 | `prompts/stages/R2-requirements-analyst.md` | `03-requirements/*` |
| 3 | R3 | `prompts/stages/R3-architect.md` | `04-architecture/*` |
| 4 | R4 | `prompts/stages/R4-skeleton-engineer.md` | 实际代码骨架 + `05-skeleton/*` |
| 5 | R5 | `prompts/stages/R5-developer.md` | 功能代码 + `06-development/FEAT-*` |
| 6 | R6 | `prompts/stages/R6-qa-engineer.md` | 测试 + `07-quality/*` |
| 7 | R7 | `prompts/stages/R7-devops-iterator.md` | 部署 + `08-iteration/*` |

#### F.3 每个阶段的典型时长

| Stage | 纯 AI 工作 | 加上思考/讨论 |
|-------|-----------|--------------|
| 0 可行性 | 20-60 分钟 | 几小时到几天 |
| 1 愿景 | 30-90 分钟 | 几天 |
| 2 需求 | 1-3 小时 | 几天到一周 |
| 3 架构 | 1-3 小时 | 几天到一周 |
| 4 骨架 | 2-6 小时 | 1-3 天 |
| 5 开发 | - | 几周到几个月 |
| 6 质量 | 几小时到几天 | 一周 |
| 7 上线 | 几小时 | 持续 |

---

### 🅶️ Step G：持续使用辅助角色（持续）

从 Stage 3 开始就可以用这些了：

#### R8 项目协调官（**高频**）

- 每个阶段结束时做门禁审查
- 每周一次做项目健康度巡检
- 遇到重大决策时求助
- **保持一个长期窗口**

#### R9 代码审查员（**Stage 5 起高频**）

- 每次完成一个功能提交前
- 新开窗口审查代码
- 发给 R9 看：代码 diff + 任务文档 FEAT-XXX

#### R10 文档守护者（**定期**）

- 每个阶段结束时做一次
- 每两周一次全面巡检
- 防止文档和代码脱节

#### R11 风险侦察兵（**按需**）

- Stage 0 后做项目风险全景
- Stage 3 后做架构风险扫描
- Stage 6 做安全和性能风险深扫
- 每月一次定期巡检

---

### 🅷️ Step H：上线后迭代（持续）

上线不是终点。继续：

- 用 R7 运维迭代官处理发布、监控
- 用 R9 审查每次代码变更
- 用 R10/R11 定期健康检查
- 每 3-6 个月用 R8 做战略性复盘

## 四、第一次使用时你会有的疑问

### Q1: "每个 Stage 都要把角色 Prompt 全文粘贴？很麻烦。"

**答**: 是的，每个角色开新对话窗口都要粘贴一次。这是必要的成本，因为它让每个角色**干净地以新身份工作**，不被其他上下文污染。

单次粘贴约 30 秒。相比跑偏后返工的时间，这完全值得。

### Q2: "Claude 回复时说它没读到文档，怎么办？"

**答**: 
- **Claude.ai 本身不能读文件**——你需要**把文档内容复制粘贴**给它
- 每个角色 Prompt 都有"启动协议"，会要求你提供哪些文档
- 按它的要求逐个粘贴即可

### Q3: "我可以跳过某些阶段吗？比如直接从 Stage 2 开始？"

**答**: 技术上可以，但不推荐。跳过 Stage 0 可能做了不该做的项目，跳过 Stage 1/2 可能做错方向。

**例外**:
- 项目已经在做了，现在只想引入这套系统管理 → 可以"补做" Stage 1-3，然后正常推进
- 非常小的项目（< 1 周）→ 可以简化，但至少做 Stage 1 + Stage 2

### Q4: "Stage 5 要开多久？会不会持续几个月？"

**答**: 对，**Stage 5 是最长的阶段**。通常占整个项目时间的 40-60%。

在这期间：
- 每个功能 = 一个 FEAT-XXX.md + 一次 R9 审查
- 保持 R8 窗口活着做协调
- 每周用 R10 巡检一次文档
- 每月用 R11 扫一次风险

### Q5: "忘了之前某个决策，怎么快速找到？"

**答**: 这正是 `DECISIONS.md` 存在的意义。打开它用 `Ctrl+F` 搜关键词。

如果 `DECISIONS.md` 里没记，但决策确实发生过 → 说明文档失职了，用 R10 做一次巡检。

### Q6: "项目成本会很高吗？"

**答**: 主要成本是 **Claude 订阅**（$20-100/月）和 **个人时间**。

如果使用 Claude API 每月大概 $10-50（看项目大小）。相比节省的时间和避免的错误，ROI 很高。

### Q7: "模型版本过期了 Prompt 还能用吗？"

**答**: 能。这套系统的 Prompt 不依赖具体模型版本，任何够强的 LLM（Claude 3.5+, GPT-4+, Gemini Pro+）都能用。推荐 Claude Opus。

### Q8: "我写到一半想改需求，怎么办？"

**答**: 这很正常，是**双向回流**机制要处理的事情。

- 如果是小修改：在当前阶段处理 + 更新 DECISIONS.md
- 如果是大调整：去找 R8 讨论"是否回退到 Stage 2 重做"
- **永远不要**偷偷改，一定要更新文档

## 五、下一步学什么

现在你已经知道如何启动和推进项目。接下来深入学习：

### 🎓 进阶阅读（在 docs/ 目录）

1. **`BEST_PRACTICES.md`** - 长期使用的最佳实践
   - 如何写好 CONSTRAINTS.md
   - 如何让 R8 给出更好的建议
   - 如何建立"压箱底"的 Prompt 调教技巧

2. **`TROUBLESHOOTING.md`** - 故障排查
   - 常见问题和解决方案
   - 角色"跑偏"时如何拉回来
   - 文档漂移时如何止损

### 📚 精读（在 .ai-workflow/docs/ 或等效位置）

3. **`PHILOSOPHY.md`** - 为什么这套系统是这样设计的
4. **`WORKFLOW_OVERVIEW.md`** - 7 阶段工作流的详细说明
5. **`ROLES.md`** - 12 角色的完整职责分工

### 💪 动手练习

第一个项目不要太大。建议：

- **练手项目**: 1-2 周能完成的小应用
- **目标**: 走完 Stage 0-7 所有阶段，体验完整流程
- **重点**: 流程熟悉 > 项目成果

这样你就内化了方法论，后续做大项目就得心应手。
