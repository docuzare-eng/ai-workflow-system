# 门禁清单 02 - 需求规格化

> Stage 2（需求规格化）完成时的验收清单。R2 自检 + R8 审查。

---

## 一、文档完整性 🔴

- [ ] `REQUIREMENTS.md` 已创建并完整
- [ ] `USER_STORIES.md` 已创建
- [ ] `USER_JOURNEYS.md` 已创建（至少覆盖 2-3 个核心旅程）

## 二、用户故事数量与质量 🔴

- [ ] P0 故事数合理（通常 10-30 个）
- [ ] 每个故事符合 "As a... I want... So that..." 格式
- [ ] 每个故事有至少 3 条 AC（成功 + 至少 2 个边界/失败）
- [ ] AC 用 Given-When-Then 格式
- [ ] 每个故事有估算（T-shirt Size）

## 三、故事粒度 🔴

- [ ] 没有 XL 级故事（应拆分）
- [ ] 每个 AC 可独立测试
- [ ] 每个故事可独立验收

## 四、覆盖完整性 🔴

### 对照 VISION.md
- [ ] 每个北极星功能有对应故事
- [ ] MVP 所有功能都有故事覆盖
- [ ] 没有超出 VISION 的"顺手加"

### 完整性
- [ ] Happy Path 覆盖
- [ ] 错误路径覆盖
- [ ] 边界情况覆盖
- [ ] 权限场景覆盖

## 五、非功能需求 🔴

- [ ] 性能需求明确（具体数字，不是"快一点"）
- [ ] 可用性需求明确
- [ ] 安全需求明确
- [ ] 兼容性需求明确
- [ ] 合规需求明确（如适用）

## 六、用户角色与权限 🔴

- [ ] 角色定义清晰
- [ ] 权限矩阵完整
- [ ] 每个操作明确哪些角色可做

## 七、数据需求 🟡

- [ ] 核心业务实体已列出
- [ ] 关键业务规则已明确
- [ ] 数据量级有预估

## 八、用户旅程 🟡

- [ ] 至少 2-3 个核心旅程有详细描绘
- [ ] 每个旅程有触点和情绪地图
- [ ] 每个旅程覆盖完整的用户目标

## 九、第三方集成 🟡

- [ ] 所有依赖的第三方服务已列出
- [ ] 每个服务的重要性和替代方案明确

## 十、范围管理 🔴

- [ ] 有"明确不做"清单（至少 3 项）
- [ ] 约束和假设明确
- [ ] 未解决的问题已列出

## 十一、可执行性 🔴

### 开发友好
- [ ] 工程师能基于故事独立开发
- [ ] 没有"等等再说"的模糊地方
- [ ] 有 UI/UX 提示（不必完整设计稿）

### 测试友好
- [ ] 每个 AC 可转化为测试用例
- [ ] 边界情况考虑充分

### 估算友好
- [ ] 故事粒度适合估算
- [ ] 依赖关系清晰

## 十二、对下一阶段的支持 🟡

- [ ] 架构师能基于需求推导架构
- [ ] 关键用例对架构的压力点已标注
- [ ] CONTEXT_HANDOFF 给 R3 已写
