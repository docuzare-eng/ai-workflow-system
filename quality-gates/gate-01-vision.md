## 一、文档完整性 🔴

- [ ] `VISION.md` 已创建并完整
- [ ] 所有占位符已填充
- [ ] `PROJECT_CONSTITUTION.md` 已根据愿景更新

## 二、愿景陈述 🔴

- [ ] 愿景陈述具体有画面（100-200 字）
- [ ] 有野心但不吹牛
- [ ] 对项目有激励作用
- [ ] 能让第三方理解

## 三、价值主张 🔴

- [ ] 一句话价值公式填写完整
- [ ] Elevator Pitch（三句话）清晰
- [ ] 非技术人员能听懂

## 四、目标用户 🔴

- [ ] 至少有 1 个详细的主要用户画像
- [ ] 画像有"典型的一天"描述
- [ ] 痛点具体（不是"有点麻烦"）
- [ ] 明确了**排除的用户**
- [ ] 替代方案（当前用户的凑合解）

## 五、差异化定位 🔴

- [ ] 至少对比了 3 个竞品/替代方案
- [ ] 差异化点明确
- [ ] 定位坐标图合理
- [ ] 占据位置有价值

## 六、MVP 范围 🔴

### 北极星功能
- [ ] 明确了"如果只能做一个，是哪个"
- [ ] 这一个功能能独立承担核心价值

### MVP 功能清单
- [ ] 功能数量合理（3-5 个）
- [ ] 每个功能都必要（不能砍掉）
- [ ] 每个功能有用户价值说明

### 明确不做
- [ ] 列出了至少 5 个"明确不做"的功能
- [ ] 每个都有理由

## 七、成长路径 🟡

- [ ] MVP 之后的 V1.0, V2.0 大致方向清晰
- [ ] 长期愿景有描绘
- [ ] 阶段之间有逻辑递进

## 八、关键场景 🔴

- [ ] 至少描述了 2-3 个关键使用场景
- [ ] 每个场景有触发、流程、结果
- [ ] 场景真实可信

## 九、核心指标 🟡

- [ ] 定义了北极星指标
- [ ] AARRR 辅助指标完整
- [ ] 指标可量化

## 十、关键假设 🔴

- [ ] 列出了至少 3 个核心假设
- [ ] 每个假设有验证方法
- [ ] 假设的优先级明确

## 十一、"砍掉的勇气" 🔴

关键检验：是否做到了"少即是多"？

- [ ] MVP 功能 ≤ 5 个
- [ ] 能回答"为什么砍掉 XX"
- [ ] 没有"面面俱到"的妥协
- [ ] 聚焦清晰

## 十二、与 Stage 0 的一致性 🔴

- [ ] 愿景与 Stage 0 的可行性分析一致
- [ ] 没有无根据的扩大范围
- [ ] 用户定位保持一致

- [ ] CONTEXT_HANDOFF 给 R2 的笔记已写
