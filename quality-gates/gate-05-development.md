# 门禁清单 05 - 功能开发

> Stage 5（功能开发）完成时的验收清单。
> 不同于其他阶段，Stage 5 的验收分**单功能**和**整体**两层。

---

## 第一部分：单个功能的验收（每个 FEAT-XXX）

### 每完成一个功能，R9 代码审查前用此清单自检。

## A.1 功能正确性 🔴

- [ ] 所有 AC 都已实现
- [ ] 所有 AC 都已测试
- [ ] 所有边界情况都已处理
- [ ] 错误场景有友好响应

## A.2 代码质量 🔴

- [ ] 符合 CONSTRAINTS.md 的代码规范
- [ ] 无 any 类型（除非必要）
- [ ] 无 console.log（用 logger）
- [ ] 无硬编码敏感信息
- [ ] 函数不超过 50 行
- [ ] 无重复代码

## A.3 架构合规 🔴

- [ ] 放置在正确的模块/目录
- [ ] 分层正确（UI 不直接访问 DB）
- [ ] 依赖方向正确
- [ ] 未引入未经决策的新依赖
- [ ] 模块通过 index.ts 暴露（不跨模块访问内部）

## A.4 安全 🔴

- [ ] 输入用 Zod 验证
- [ ] SQL 参数化
- [ ] 权限检查到位
- [ ] 敏感日志脱敏
- [ ] 错误信息不泄露堆栈

## A.5 测试 🔴

- [ ] 单元测试覆盖核心逻辑
- [ ] 至少包含 Happy Path + 错误路径
- [ ] 测试通过
- [ ] 覆盖率不下降

## A.6 文档 🔴

- [ ] 任务文档 FEAT-XXX.md 更新
- [ ] 如涉及 API 变更，API_DESIGN.md 更新
- [ ] 如涉及数据库变更，DATABASE.md 更新
- [ ] 复杂逻辑有代码注释

## A.7 提交 🔴

- [ ] Commit message 规范（feat/fix/chore 等）
- [ ] Commit 粒度合理（不大不小）
- [ ] PR 描述清晰
