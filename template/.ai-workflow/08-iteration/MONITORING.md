## 一、监控策略

### 四个黄金信号 (Four Golden Signals)

1. **延迟 (Latency)**: 请求耗时
2. **流量 (Traffic)**: 请求量
3. **错误 (Errors)**: 错误率
4. **饱和度 (Saturation)**: 资源使用率

### 监控层次

```
┌──────────────────────┐
│  业务指标            │  用户注册数、转化率等
├──────────────────────┤
│  应用指标            │  API 延迟、错误率
├──────────────────────┤
│  系统指标            │  CPU、内存、网络
├──────────────────────┤
│  基础设施            │  云服务可用性
└──────────────────────┘
```

## 二、关键指标 (KPIs)

### 技术指标

| 指标 | 目标 | 告警阈值 |
|------|------|---------|
| 可用性 | 99.9% | < 99.5% |
| P95 延迟 | < 500ms | > 1s |
| 错误率 | < 0.1% | > 1% |
| CPU | < 70% | > 85% |
| 内存 | < 80% | > 90% |
| DB 连接池 | < 70% | > 85% |

### 业务指标

| 指标 | 监控频率 | 异常定义 |
|------|---------|---------|
| 新注册 | 每小时 | 同比下降 > 50% |
| 登录成功率 | 实时 | < 95% |
| 核心功能转化 | 每日 | 下降 > 30% |
| 支付成功率 | 实时 | < 98% |

## 三、告警配置

### 告警分级

| 级别 | 响应时间 | 通知方式 |
|------|---------|---------|
| P0 (紧急) | < 5 分钟 | 电话 + 短信 + 微信 |
| P1 (高) | < 30 分钟 | 短信 + 微信 |
| P2 (中) | < 2 小时 | 微信 + 邮件 |
| P3 (低) | 下个工作日 | 邮件 |

### 告警规则示例

```yaml
# 错误率飙升
- name: high_error_rate
  condition: error_rate > 1% for 5min
  severity: P1
  notify: [on-call]

# 服务宕机
- name: service_down
  condition: availability < 90% for 1min
  severity: P0
  notify: [on-call, escalation]

# 慢查询
- name: slow_queries
  condition: p95 > 1s for 10min
  severity: P2
  notify: [team]
```

### 告警疲劳防治

- ❌ 告警过多失去意义
- ✅ 只告警真正需要人工介入的
- ✅ 定期 review 告警合理性
- ✅ 合并相似告警

## 四、日志策略

### 日志分级

- `DEBUG`: 开发调试（生产不记录）
- `INFO`: 正常业务事件
- `WARN`: 可能的问题
- `ERROR`: 错误（需关注）
- `FATAL`: 严重错误（需立即响应）

### 日志内容规范

**必含字段**:
```json
{
  "timestamp": "ISO8601",
  "level": "info",
  "service": "api",
  "requestId": "req_xxx",
  "userId": "optional",
  "event": "user.login",
  "message": "...",
  "metadata": {}
}
```

### 敏感信息处理

- 密码: 绝不记录
- Token: 只记录前 8 位
- 邮箱: 脱敏（a***@example.com）
- 身份证等: 脱敏

### 日志聚合

- **工具**: [Axiom / Datadog / 自建 ELK]
- **保留**: 30 天（一般） / 1 年（安全）
- **搜索**: 支持按 requestId 全链路查询

## 五、错误追踪

### 工具: [Sentry / 其他]

### 配置

- 前后端都接入
- 源码 Sourcemap 上传（方便追错）
- 用户信息脱敏后上报
- 按 Release 版本关联

### 处理流程

```
错误上报
  ↓
自动分组（相似错误合并）
  ↓
分配责任人
  ↓
评估影响
  ↓
修复
  ↓
关闭
```

## 七、可观测性三大支柱

### 1. Logs（日志）
"发生了什么"

### 2. Metrics（指标）
"发生了多少"

### 3. Traces（追踪）
"从哪到哪"

三者结合才能快速定位问题。

## 十、事后复盘 (Postmortem)

### 复盘原则

- **无责备 (Blameless)**: 对事不对人
- **事实为基**: 基于数据和日志
- **行动为果**: 产出具体改进

### 复盘模板

```markdown

## 事故概要
- 开始时间:
- 结束时间:
- 影响:
- 级别:

[按时间顺序描述]

## 根本原因
[5 Whys 分析]

## 做得好的
[...]

## 改进项
- [ ] 改进 1
- [ ] 改进 2
```
