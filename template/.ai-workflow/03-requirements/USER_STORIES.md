## 一、故事编号与优先级规则

### 编号规则
- 格式: `US-XXX`（US = User Story）
- 编号递增，不复用（删除的故事不填补编号）
- 模块内编号可按模块前缀，如 `US-AUTH-001`

### 优先级定义

| 优先级 | 含义 | 特征 |
|--------|------|------|
| **P0** | Must Have | 不做就无法上线 MVP |
| **P1** | Should Have | 应做，MVP 后紧跟 |
| **P2** | Could Have | 锦上添花，视情况而定 |
| **P3** | Won't Have | 明确不在当前 MVP |

### 估算规则 (T-shirt Size)

| Size | 估算 | 描述 |
|------|------|------|
| XS | 几小时 | 简单修改、小配置 |
| S | 半天-1天 | 简单功能、小组件 |
| M | 2-3 天 | 常规功能 |
| L | 一周 | 较复杂功能 |
| XL | > 一周 | **警告：应拆分！** |

## 六、故事粒度检查

### 是否足够小？

每个故事应该：
- ✅ 可以在 1 周内完成（估 L 及以下）
- ✅ 可以独立部署
- ✅ 可以独立测试

如果发现故事过大（XL），按以下方式拆分：
- 按用户操作拆：注册 / 验证邮箱 / 首次登录 分开
- 按场景拆：桌面版 / 移动版 分开
- 按成功路径拆：Happy Path / Error Handling 分开

### 是否足够独立？

- ✅ 故事间依赖清晰（通过 ID 引用）
- ✅ 独立的技术任务尽量不混在同一个用户故事中
- ❌ 避免："实现用户模块" 这种大杂烩
