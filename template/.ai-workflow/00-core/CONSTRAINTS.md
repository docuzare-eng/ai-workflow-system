# 硬约束清单 (Hard Constraints)

> 本文件定义项目的**不可逾越的边界**。所有 AI 角色必须严格遵守。
> 违反约束 = 产出无效。发现约束不合理时，可以提议修改，但修改前必须遵守。

---

## 📖 约束的类型

### 🔴 硬约束（Hard Constraints）
**绝对不可违反**。违反即视为错误产出，必须返工。

### 🟡 强偏好（Strong Preferences）
**默认遵守**。违反需要明确理由并记录到 DECISIONS.md。

### 🟢 软建议（Soft Guidelines）
**参考遵循**。可根据情况灵活处理。

## 二、代码规范约束（Code Standards）

### 🔴 硬约束

1. **类型安全**
   - TypeScript `strict` 模式必须开启
   - 禁止使用 `any`（除非有注释说明不可避免的原因）
   - 所有函数必须有明确的返回类型

2. **错误处理**
   - 异步操作必须有错误处理
   - API 调用必须处理网络错误、超时、业务错误
   - 用户输入必须验证

3. **命名规范**
   - 变量/函数: `camelCase`
   - 类/类型/接口: `PascalCase`
   - 常量: `UPPER_SNAKE_CASE`
   - 文件名: `[按项目规范]`

4. **注释规范**
   - 公共 API 必须有 JSDoc
   - 复杂逻辑必须有"为什么"的注释（不是"是什么"）
   - TODO 必须带上下文和追踪方式

### 🟡 强偏好
- 单个函数不超过 50 行
- 单个文件不超过 300 行
- 嵌套层级不超过 4 层
- 函数参数不超过 4 个（多了用对象）

## 五、安全约束（Security）

### 🔴 硬约束（不可违反）

1. **敏感信息**
   - 🚨 禁止将密钥/密码硬编码到代码中
   - 🚨 禁止将 `.env` 文件提交到 Git
   - 🚨 禁止在日志中打印敏感信息（密码、令牌、信用卡）

2. **认证授权**
   - 所有非公开接口必须验证身份
   - 用户密码必须哈希存储（bcrypt / argon2）
   - Token 必须设置合理的过期时间

3. **输入输出**
   - 所有用户输入必须验证和消毒
   - SQL 查询必须使用参数化（防注入）
   - 渲染用户内容必须转义（防 XSS）

4. **依赖安全**
   - 新增依赖前必须检查 npm audit
   - 定期（至少每月）更新依赖
   - 锁定依赖版本（lockfile 提交）

### 🟡 强偏好
- HTTPS 强制（生产环境）
- CSP 头部配置
- Rate limiting 默认启用

## 八、工作流约束（Workflow）

### 🔴 硬约束

1. **Git 规范**
   - Commit message 遵循 Conventional Commits
   - 禁止直接 push 到 `main` 分支（除紧急修复）
   - 所有变更必须通过 PR
   - PR 必须经过代码审查（R9 或人工）

2. **文档同步**
   - 架构变更必须同步更新 `ARCHITECTURE.md`
   - 新增依赖必须更新 `TECH_STACK.md`
   - 决策变更必须新增 `DECISIONS.md` 条目

3. **测试要求**
   - 核心业务逻辑必须有单元测试
   - API 端点必须有集成测试
   - 关键用户流程必须有 E2E 测试

### 🟡 强偏好
- 测试覆盖率 > 70%
- 每个 PR 不超过 500 行变更
- 每日至少一次 commit

## 九、AI 协作约束（AI Rules）

### 🔴 硬约束（所有 AI 角色必须遵守）

1. **严守角色边界**
   - 不做超出当前角色职责的事
   - 发现越权需求时，建议切换到正确角色

2. **文档即真相**
   - 只基于文档做决策，不基于"我记得"
   - 任何"记忆"必须能在文档中找到出处

3. **疑问必问**
   - 信息不足时，明确提问而非假设
   - 不做"我猜你想要"的事

4. **输出规范**
   - 代码必须能运行
   - 文档必须结构化
   - 决策必须有理由

5. **安全红线**
   - 不生成含安全漏洞的代码
   - 不绕过安全约束
   - 不硬编码敏感信息

## 十、约束变更流程

当发现某条约束不合理时：

1. **不要私自违反**，先标记为"待讨论"
2. **提出变更提议**，说明：
   - 当前约束是什么
   - 为什么不合理
   - 建议如何修改
   - 影响范围
3. **经过决策**后，在 `DECISIONS.md` 中记录
4. **更新本文件**，在变更日志中记录
5. **通知 R10 文档守护者**进行一致性检查
